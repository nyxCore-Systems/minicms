# Building a Dynamic CMS: From Static Pages to Database-Driven Content

What started as a simple static site has evolved into a full-featured content management system. Today, I'm sharing the journey of implementing media uploads with Cloudinary, tackling tricky authentication bugs, and converting static product pages to database-driven content.

## The Feature Sprint: What Got Built

### 🖼️ Media Management with Cloudinary

The centerpiece of this development session was building a complete media management system. Here's what went into it:

**Backend Infrastructure:**
- Cloudinary SDK integration for cloud storage
- RESTful API routes for upload, list, and delete operations  
- Prisma schema extensions with `Media` model and `MediaType` enum

**Admin Interface:**
- Drag-and-drop gallery with visual file management
- Filter tabs for organizing different media types
- Modal image picker dialog for content integration
- Inline markdown image insertion directly from the editor

The result? Content creators can now upload images once and reuse them across any page, with automatic cloud optimization and CDN delivery.

### 🔐 Authentication Battle Scars

Moving to Next.js 15 introduced some unexpected authentication challenges that turned into valuable learning experiences.

**The Silent Failure:**
```typescript
// This stopped working silently in Next.js 15
const session = await getServerSession(authOptions);
```

The culprit? Next.js 15's async `headers()` and `cookies()` functions don't play nicely with NextAuth's `getServerSession`. The solution required switching to JWT token validation:

```typescript
// The working approach
import { getToken } from "next-auth/jwt";
import { cookies } from "next/headers";

const token = await getToken({ 
  req: { cookies: Object.fromEntries(cookies()) } 
});
```

### 🌐 Production Environment Gotchas

Deploying to production revealed several configuration issues that only surface in live environments:

**HTTPS vs HTTP Session Cookies:**
When changing `NEXTAUTH_URL` from HTTP to HTTPS, session cookie names change from `next-auth.session-token` to `__Secure-next-auth.session-token`. Existing users had to log out and back in for the change to take effect.

**API Key Validation:**
Cloudinary's API keys are strictly numeric. A malformed key like `k1Ftv47RvoOJV3D6oUVNJBKYlf8` fails silently with 401 errors, while the correct format `747574658755142` works perfectly.

## Lessons Learned

### Authentication in Modern Next.js

**The Problem:** NextAuth's `getServerSession` fails silently in Next.js 15 server components due to async context changes.

**The Solution:** Use `getToken` from `next-auth/jwt` with manual cookie extraction. While more verbose, it's reliable across all Next.js versions.

**The Takeaway:** Always test authentication in production environments. Silent failures in auth can leave your admin routes completely unprotected.

### Environment Configuration Complexity

**The Problem:** Different cookie security models between development (HTTP) and production (HTTPS) environments.

**The Solution:** Plan for session invalidation when promoting HTTPS. Document the logout requirement for existing users.

**The Takeaway:** Environment configuration changes often have user-facing impacts that aren't immediately obvious.

### Third-Party Service Integration

**The Problem:** Invalid API credentials can cause cryptic failures that are hard to debug.

**The Solution:** Validate service credentials early and add comprehensive error handling.

**The Takeaway:** When integrating cloud services, always test with production credentials in a staging environment first.

## Technical Architecture Highlights

The resulting system uses a clean separation of concerns:

- **Data Layer:** Prisma ORM with PostgreSQL for structured content and media metadata
- **Storage Layer:** Cloudinary for media files with automatic optimization
- **API Layer:** Next.js 13+ app router with protected admin routes
- **UI Layer:** React Server Components with progressive enhancement

The admin interface feels native to the platform while the public pages maintain excellent performance through static generation where possible.
