# Building a Dynamic CMS: From Hidden FAQs to Live SEO Indicators

Yesterday was one of those development days where everything clicks into place. What started as a simple bug fix turned into a comprehensive CMS overhaul that transformed how our knife retailer client manages their content and SEO strategy.

## The Challenge: When Static Meets Dynamic

Our client's website had grown organically—some pages were hardcoded, others pulled from a database, and the admin interface was a patchwork of different approaches. The immediate problem? FAQ items were mysteriously disappearing from the editor after SEO generation. But as we dug deeper, we uncovered a bigger opportunity.

## The Detective Work: Solving the Invisible FAQ Mystery

The FAQ bug was our entry point. After SEO generation, editors would lose visibility into their carefully crafted FAQ sections. The root cause was simple but frustrating: the SEO panel was stealing focus, and there was no easy way to preview the generated FAQs.

**The Solution:** We added a dedicated FAQ preview section to the SEO panel in the content editor:

```typescript
// src/app/admin/content/[id]/page.tsx
// Auto-expand SEO panel after generation
setShowSeo(true);

// New FAQ preview section renders alongside SEO controls
{faqs && faqs.length > 0 && (
  <div className="faq-preview-section">
    {faqs.map((faq, index) => (
      <div key={index} className="faq-item">
        <h4>{faq.question}</h4>
        <p>{faq.answer}</p>
      </div>
    ))}
  </div>
)}
```

Now editors can see their FAQs immediately after generation, eliminating the guesswork.

## The Bigger Picture: Real-Time SEO Status Tracking

While fixing the FAQ issue, we realized content managers had no visibility into which pages had SEO optimization and which didn't. They were flying blind.

We built a new client-side component that transforms the admin content list into a SEO command center:

```typescript
// src/components/admin/ContentListClient.tsx
const ContentListClient = ({ initialContent }) => {
  return (
    <div className="content-grid">
      {content.map(item => (
        <div key={item.id} className="content-item">
          {/* Visual SEO status indicator */}
          <div className={`status-dot ${item.seoTitle ? 'green' : 'red'}`} />
          
          {/* Inline SEO generation */}
          {!item.seoTitle && (
            <button onClick={() => generateSEO(item.id)}>
              SEO generieren
            </button>
          )}
        </div>
      ))}
    </div>
  );
};
```

The result? Content managers can now see at a glance which pages need SEO attention and generate it with a single click—no more diving into individual pages to check SEO status.

## Lessons Learned: The Developer's Survival Kit

Every development session teaches you something. Here's what saved us time (and sanity):

### Context Overflow Management
Working with AI assistants across long sessions means context gets compressed. We learned to maintain detailed session handoff documents—essentially "letters to our future selves" that preserve critical state information.

### The Template Literal Trap
When running database scripts with `npx tsx -e`, template literals containing `$disconnect()` cause esbuild syntax errors. **Workaround**: Write temporary `.ts` files instead of inline execution. It's more steps but infinitely more reliable.

### Cache Invalidation Blues
After multi-file changes, Next.js sometimes gets confused with module resolution errors like `Cannot find module './4996.js'`. **Nuclear option**: `rm -rf .next` and rebuild. Sometimes the simplest solution is the best solution.
