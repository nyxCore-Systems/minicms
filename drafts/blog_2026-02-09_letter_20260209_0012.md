# Debugging the Double-Stringify Trap: A Production TypeError Mystery

Nothing quite gets your heart racing like a production error that seems to come out of nowhere. Last week, our Next.js application started throwing cryptic `TypeError: .map() is not a function` errors, and what began as a simple bug hunt turned into a fascinating deep-dive into the subtle dangers of JSON serialization with Prisma.

## The Mystery Begins

The error was maddeningly vague. Production logs showed a TypeError originating from a minified chunk (`1747.js`), making it nearly impossible to trace the exact location. Users were seeing "Application error" pages instead of our carefully crafted content, and the minified stack traces weren't giving us much to work with.

Our first instinct was to suspect the usual culprits—maybe our breadcrumb component was receiving unexpected data, or perhaps a recent API change had broken our assumptions about data structures. But as we dug deeper, the real culprit emerged from an unexpected corner of our codebase.

## The Root Cause: A Double-Stringify Nightmare

The breakthrough came when we decided to query our database directly and examine the `faqSchema` field that was supposed to contain an array of FAQ items. What we discovered was both illuminating and embarrassing:

```javascript
// What we expected in the database:
faqSchema: [
  { question: "How does it work?", answer: "Like this..." },
  { question: "Is it safe?", answer: "Absolutely..." }
]

// What we actually found:
faqSchema: "[{\"question\":\"How does it work?\",\"answer\":\"Like this...\"},{\"question\":\"Is it safe?\",\"answer\":\"Absolutely...\"}]"
```

Seven out of ten pages in our database had their `faqSchema` field stored as a stringified string rather than a proper JSON array. When our React components tried to call `.map()` on what they expected to be an array, they were actually trying to call it on a string—hence the TypeError.

## The Double-Stringify Trap Explained

The root cause was a classic case of double-serialization. Here's how it happened:

```javascript
// ❌ The problematic code
const response = await fetch(`/api/admin/pages/${id}`, {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    seoData: JSON.stringify(seoResult), // First stringify
    faqSchema: JSON.stringify(seoResult.faqItems) // First stringify
  })
})

// Inside the API handler, this gets parsed back to:
// { seoData: "stringified-json", faqSchema: "stringified-json" }
// Then stored directly in Prisma's Json field
```

The problem was that we were calling `JSON.stringify()` on our data before passing it to `fetch()`, which then called `JSON.stringify()` again on the entire body. This double-encoding turned our arrays into stringified strings.

## The Fix: Defensive Programming and Clean Data Flow

We implemented a three-pronged solution:

### 1. Remove Double Stringification

```javascript
// ✅ The corrected code
const response = await fetch(`/api/admin/pages/${id}`, {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    seoData: seoResult, // Let fetch handle the stringification
    faqSchema: seoResult.faqItems // Let fetch handle the stringification
  })
})
```

### 2. Add Defensive Parsing

We created a utility function to handle both correct and corrupted data:

```javascript
function parseFaqSchema(faqSchema) {
  if (!faqSchema) return [];
  if (Array.isArray(faqSchema)) return faqSchema;
  if (typeof faqSchema === 'string') {
    try {
      const parsed = JSON.parse(faqSchema);
      return Array.isArray(parsed) ? parsed : [];
    } catch {
      return [];
    }
  }
  return [];
}
```

### 3. Database Migration

We wrote a one-time script to clean up the corrupted data:

```javascript
// Migration script to fix double-stringified data
const pages = await prisma.page.findMany();
for (const page of pages) {
  if (typeof page.faqSchema === 'string') {
    const fixed = JSON.parse(page.faqSchema);
    await prisma.page.update({
      where: { id: page.id },
      data: { faqSchema: fixed }
    });
  }
}
```

## Lessons Learned

This debugging session taught us several valuable lessons:

### 1. **Prisma Json Fields Are Sensitive**
Prisma's `Json` field type stores data exactly as you provide it. If you pass a string, it stores a string. If you pass an object, it stores an object. There's no automatic parsing magic.

### 2. **Never Double-Stringify**
When using `fetch()` with JSON data, remember that `JSON.stringify(body)` will serialize everything in the body object. Don't pre-stringify individual fields unless you specifically want them stored as strings.

### 3. **Minified Errors Are Debugging Nightmares**
Having a good logging strategy for production is crucial. Source maps and proper error boundaries can save hours of detective work.

### 4. **Defensive Programming Pays Off**
Adding type checks and fallbacks for critical data structures can prevent user-facing errors even when underlying data gets corrupted.

## The Happy Ending

After deploying our fixes, we verified all 12 production pages manually. Every single one returned HTTP 200 with proper FAQ JSON-LD structured data where expected. The `TypeError: .map() is not a function` errors disappeared completely.

More importantly, we now have safeguards in place to prevent this type of issue from recurring, and our debugging skills got a solid workout in the process.

## Key Takeaways for Developers

1. **Be explicit about data types** when working with Prisma Json fields
2. **Avoid double-serialization** in API calls—let `fetch()` handle the final JSON.stringify()
3. **Implement defensive parsing** for critical data structures
4. **Test with realistic data** that might include edge cases or corrupted states
5. **Keep detailed debugging notes**—they often reveal patterns that aren't obvious in the moment

Sometimes the most valuable lessons come from the messiest bugs. This particular adventure reminded us that even experienced developers can fall into subtle traps, and that methodical debugging combined with defensive programming practices can turn frustrating errors into learning opportunities.
