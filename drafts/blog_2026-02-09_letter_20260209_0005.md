# Building a Modern Admin Panel: Dark Mode, Dynamic Menus, and Advanced Markdown in Next.js

Recently, I wrapped up an intensive development session building out a comprehensive admin panel for a Next.js application. What started as a simple setup page evolved into a full-featured content management system with some pretty interesting technical challenges. Here's the story of how it all came together.

## The Mission: More Than Just CRUD

The goal was ambitious: create an admin interface that could handle site settings, implement a proper dark mode, build a dynamic database-driven menu system, and enhance markdown parsing with custom directives and image sizing. Oh, and make it all work seamlessly together.

By the end of the session, everything was complete, committed (`7bbdd99`), and deployed to production at [landingpage-e-ventschau.vercel.app](https://landingpage-e-ventschau.vercel.app).

## The Architecture: Database-First Design

### Dynamic Menu System

One of the most interesting challenges was building a truly dynamic menu system. Instead of hardcoded navigation, I implemented a self-referencing database structure using Prisma:

```typescript
// Prisma schema for nested menu items
model MenuItem {
  id        String     @id @default(cuid())
  title     String
  url       String?
  parentId  String?
  order     Int        @default(0)
  parent    MenuItem?  @relation("MenuHierarchy", fields: [parentId], references: [id])
  children  MenuItem[] @relation("MenuHierarchy")
  createdAt DateTime   @default(now())
  updatedAt DateTime   @updatedAt
}
```

This allows for unlimited nesting levels and makes the menu completely manageable through the admin interface. The real magic happens in the header component, which was split into server and client components to handle the database fetching efficiently.

## Enhanced Markdown: Beyond Basic Parsing

The markdown system deserved special attention. Rather than using a basic parser, I built a custom solution that supports:

### Nested Directives
Custom blocks using `:::` syntax that can be nested at any depth:

```markdown
::: callout
This is a callout box
::: feature
  Nested feature box inside!
:::
Back to the callout
:::
```

### Column Layouts
Dynamic column splitting with `---` separators that respects nesting:

```markdown
::: columns-2
Left column content
---
Right column content
:::
```

### Smart Image Sizing
Images with size specifications in the alt text:

```markdown
![Beautiful landscape|600x400](image.jpg)
![Portrait|300](portrait.jpg)
```

The parsing logic uses a depth-aware algorithm that properly handles nested blocks and maintains context throughout the parsing process.

## Dark Mode: The Right Way

Implementing dark mode properly meant more than just toggling a CSS class. The solution involved:

1. **Tailwind Configuration**: Setting up `darkMode: 'class'` in the config
2. **CSS Variables**: Defining color schemes for both light and dark modes
3. **Anti-Flash Script**: Preventing the dreaded flash of wrong theme on page load
4. **Persistent State**: Storing user preferences across sessions

The anti-flash script in the root layout ensures users never see a jarring theme change:

```javascript
// Inline script in layout.tsx prevents flash
(function() {
  try {
    const theme = localStorage.getItem('theme')
    if (theme === 'dark' || (!theme && window.matchMedia('(prefers-color-scheme: dark)').matches)) {
      document.documentElement.classList.add('dark')
    }
  } catch (e) {}
})()
```

## API Design: RESTful and Intuitive

The API structure follows REST principles with logical endpoints:

- `GET /api/menu` - Public menu fetching
- `GET|PUT /api/admin/settings` - Site settings management
- `GET|POST /api/admin/menu` - Menu collection operations
- `GET|PUT|DELETE /api/admin/menu/[id]` - Individual menu item operations

Each endpoint handles proper error cases and returns consistent response formats.

## Lessons Learned: The Real World Strikes Back

### Database Cold Starts
**The Problem**: Immediately curling `/api/menu` after dev server start resulted in `PrismaClientInitializationError: Can't reach database server`.

**The Reality**: Neon (our serverless database) needs a warm-up period. This isn't a bug—it's the nature of serverless databases.

**The Solution**: Implement retry logic and expect a few seconds of cold start time.

### Cache Invalidation Chaos
**The Problem**: After extensive file modifications, navigating to `/admin/content` threw `ENOENT: no such file or directory, .next/server/vendor-chunks/next-auth.js`.

**The Reality**: Next.js's `.next` cache can become stale after major multi-file changes.

**The Solution**: When in doubt, `rm -rf .next` and restart the dev server. It's the "turn it off and on again" of Next.js development.

### Shell Glob Gotchas
**The Problem**: Git commands with unquoted brackets and parentheses (`[id]`, `(public)`) caused zsh glob expansion errors.

**The Reality**: Shell metacharacters need proper quoting in file paths.

**The Solution**: Always quote special characters: `"src/app/admin/content/[id]/page.tsx"`.
