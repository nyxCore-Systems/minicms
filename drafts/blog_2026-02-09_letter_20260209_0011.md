# When JSON.stringify() Strikes Twice: A Production Bug Story

Nothing quite gets your attention like a production error that's seemingly impossible to debug. Today, I want to share the story of how a single extra `JSON.stringify()` call brought down our FAQ sections across multiple pages, and the detective work that led us to the root cause.

## The Problem: A Mysterious `.map() is not a function`

It started with a TypeError in production: `.map() is not a function`. The error was coming from minified JavaScript (`1747.js`), making it nearly impossible to trace back to the source code. Users were seeing broken pages, and our monitoring showed HTTP errors across multiple routes.

The error seemed to be related to our FAQ schema rendering, but the stack trace pointed to minified chunks that gave us no meaningful context about what was actually failing.

## The Investigation: Following False Leads

Like many debugging sessions, we started down the wrong path. The initial assumption was that our Breadcrumbs component was somehow corrupting the FAQ data. We spent considerable time examining component props and data flow, only to realize we were barking up the wrong tree.

The real breakthrough came when we decided to query the database directly to examine the `faqSchema` field across all our pages:

```javascript
// Checking the actual data types in our Prisma Json fields
const pages = await prisma.page.findMany({
  select: { id: true, slug: true, faqSchema: true }
});

pages.forEach(page => {
  console.log(`${page.slug}: ${typeof page.faqSchema}`);
});
```

What we discovered was shocking: 7 out of 10 pages had `faqSchema` stored as strings instead of arrays. But how does a JSON array become a string in a Prisma `Json` field?

## The Root Cause: Double Stringification

The culprit was hiding in our admin panel's save functionality. Here's what was happening:

```javascript
// ❌ The problematic code
const body = {
  // ... other fields
  faqSchema: JSON.stringify(faqSchema), // First stringification
  seoData: JSON.stringify(seoData)      // Same issue here
};

// Then in the fetch call
const response = await fetch('/api/admin/pages/[id]', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(body) // Second stringification!
});
```

The `fetch` API automatically handles JSON serialization when you call `JSON.stringify()` on the body. By manually stringifying our fields first, we were causing **double stringification**.

Instead of storing this in the database:
```json
[{"question": "What is this?", "answer": "An FAQ item"}]
```

We were storing this:
```json
"[{\"question\": \"What is this?\", \"answer\": \"An FAQ item\"}]"
```

When our React components tried to call `.map()` on what they expected to be an array, they were actually trying to call it on a string—hence the TypeError.

## The Fix: Clean Up and Safeguard

The solution involved three key changes:

### 1. Remove the Extra Stringification

```javascript
// ✅ Fixed version
const body = {
  // ... other fields
  faqSchema: faqSchema, // Let fetch handle the serialization
  seoData: seoData
};

const response = await fetch('/api/admin/pages/[id]', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(body) // Only stringify the entire body
});
```

### 2. Add Defensive Parsing

We created a helper function to safely handle both string and array types from our database:

```javascript
function parseFaqSchema(faqSchema) {
  if (!faqSchema) return [];
  if (Array.isArray(faqSchema)) return faqSchema;
  if (typeof faqSchema === 'string') {
    try {
      const parsed = JSON.parse(faqSchema);
      return Array.isArray(parsed) ? parsed : [];
    } catch {
      return [];
    }
  }
  return [];
}
```

### 3. Data Migration

We ran a script to fix the corrupted data in our database, converting the double-stringified JSON back to proper arrays.

## Lessons Learned

This debugging session taught us several valuable lessons:

1. **Never double-stringify**: When using `fetch` with JSON, let the `JSON.stringify(body)` call handle all serialization. Don't pre-stringify individual fields.

2. **Test with real data**: This bug only manifested with pages that had been saved through our admin interface. Fresh data worked fine, which delayed our discovery.

3. **Minified stack traces are your enemy**: Consider using source maps in production, or at least have a way to reproduce errors locally with unminified code.

4. **Database inspection is powerful**: When components behave unexpectedly, sometimes the fastest path to answers is examining the raw data.

5. **Defensive programming pays off**: Our new parsing helper ensures this type of data corruption won't break the UI, even if it happens again.

## The Result

After deploying the fix (commit `70f37e5`), all pages returned to HTTP 200 status, and FAQ sections rendered correctly across the entire site. The production error rate dropped to zero, and our users could once again access all content without issues.

## Key Takeaway

The moral of this story? **Respect the JSON serialization boundary**. When you're sending data via `fetch` with `Content-Type: application/json`, trust the API to handle the serialization. Manual `JSON.stringify()` calls on individual fields are almost always a mistake and can lead to subtle bugs that are painful to debug in production.
