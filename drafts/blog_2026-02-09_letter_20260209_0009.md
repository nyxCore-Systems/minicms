# Building Resilient Web Apps: Database Retries, Smart Caching, and SEO Automation

Yesterday was one of those development sessions where everything that could go wrong did go wrong—and yet, by the end, we had built some genuinely robust solutions. Here's the story of refactoring a knife retailer's website while learning valuable lessons about database resilience, SEO automation, and the perils of concurrent development.

## The Mission: Four Critical Improvements

We started with a clear goal: refactor the verkaufen (selling) pages to match our product page styling, eliminate server-side errors that were plaguing production, audit SEO across all pages, and build a smart system to detect when content changes make SEO metadata outdated.

Sounds straightforward, right? Well...

## Challenge #1: When Databases Go Silent

The first major hurdle came from our cloud database provider, Neon. We were experiencing intermittent connection failures—the classic `ETIMEDOUT`, `ECONNRESET`, and "socket hang up" errors that make production monitoring alerts go crazy.

Rather than just hoping the problem would go away, we built a proper solution:

```typescript
// src/lib/prisma.ts
async function withRetry<T>(
  operation: () => Promise<T>, 
  maxRetries: number = 2
): Promise<T> {
  for (let i = 0; i <= maxRetries; i++) {
    try {
      return await operation();
    } catch (error) {
      if (i === maxRetries || !isRetryableError(error)) {
        throw error;
      }
      // Progressive backoff: 500ms, then 1000ms
      await new Promise(resolve => setTimeout(resolve, 500 * (i + 1)));
    }
  }
}
```

This utility now wraps all our critical database operations in `getTenant()`, `getContent()`, `getMenuItems()`, and `getSiteSettings()`. The beauty is in its simplicity—it only retries on network-level failures, not logical errors like constraint violations.

## Challenge #2: Graceful Degradation is Everything

While fixing the retry logic, we discovered another issue: when database calls failed, they were taking down entire page renders. The solution was implementing proper fallback chains:

```typescript
// Before: Throwing errors crashed the page
export async function getContent(slug: string) {
  const page = await withRetry(() => 
    prisma.page.findFirst({ where: { slug } })
  );
  if (!page) throw new Error('Page not found');
  return page;
}

// After: Graceful fallbacks keep the site running
export async function getContent(slug: string) {
  try {
    const page = await withRetry(() => 
      prisma.page.findFirst({ where: { slug } })
    );
    return page;
  } catch (error) {
    console.error('Database error:', error);
    return null; // Let the UI handle the missing data gracefully
  }
}
```

Now when `generateMetadata()` can't fetch content, it falls back to default values instead of showing users a 500 error page.

## Challenge #3: Smart SEO State Management

Here's where things got interesting. We wanted to solve a common CMS problem: how do you know when your carefully crafted SEO metadata becomes stale after content changes?

Our solution was elegantly simple—content hashing:

```typescript
// src/lib/hash.ts - djb2 hash algorithm, base-36 encoded
function simpleHash(str: string): string {
  let hash = 5381;
  for (let i = 0; i < str.length; i++) {
    hash = ((hash << 5) + hash) + str.charCodeAt(i);
  }
  return (hash >>> 0).toString(36);
}
```

Every time content is saved, we compute a hash and store it alongside the SEO data. Now our admin interface can show three distinct states:

- 🟢 **Green dot**: SEO is current (content hash matches)
- 🟡 **Yellow dot**: SEO exists but content has changed (hash mismatch)  
- 🔴 **Red dot**: No SEO data exists

The UI adapts accordingly—showing "SEO generieren" (generate SEO) for red dots and "SEO aktualisieren" (update SEO) for yellow ones.

## Lessons Learned: The Pain Points That Teach Us

### Lesson 1: Never Run Dev Servers During Automated Edits

The most painful debugging session came from trying to run `npm run dev` while automated tools were editing files. The result? Corrupted webpack caches, mysterious module resolution errors, and cryptic messages like `Cannot find module './4996.js'`.

**The fix:** Always stop development servers before automated refactoring. A simple `rm -rf .next` cleared the corrupted cache, but we lost an hour to this gotcha.

### Lesson 2: Cold Start Database Behavior is Real

We discovered that Neon databases sometimes return empty results on the first query after a cold start. This isn't a bug—it's just how serverless databases work. But it perfectly validated our retry logic implementation.

### Lesson 3: Environment Variable Consistency Matters

We found hardcoded references to `NEXTAUTH_URL` scattered across SEO utilities, sitemaps, and robots.txt generation. Consolidating to `NEXT_PUBLIC_SITE_URL` with a sensible default (`https://e-ventschau.de`) eliminated a whole class of deployment issues.

## Key Takeaways for Your Next Project

1. **Build retry logic early**: Network failures are inevitable in cloud environments
2. **Design for graceful degradation**: Never let a single failing component crash the entire experience  
3. **Automate staleness detection**: Content and metadata drift apart—build systems to catch it
4. **Separate development and automation**: Don't run dev servers during automated code changes
5. **Consolidate configuration**: Environment variables should have single sources of truth

Building resilient web applications isn't just about handling the happy path—it's about gracefully managing all the ways things can go wrong. Yesterday's session reminded us that the best debugging sessions are the ones that result in systems becoming more robust than they were before.
