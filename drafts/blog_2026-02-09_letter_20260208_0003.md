# From Filesystem to Full-Stack: Building a Knife Marketplace with Next.js 15

After three intense development sessions, I've successfully deployed a complete knife marketplace platform to production. What started as a simple landing page has evolved into a full-featured e-commerce site with a custom CMS, markdown editor, and PostgreSQL backend. Here's the story of how it all came together - and the unexpected pitfalls that taught me valuable lessons about modern deployment workflows.

## The Vision: More Than Just a Static Site

The goal was ambitious but clear: build a comprehensive knife marketplace featuring:
- **Public storefront** with product pages, vendor profiles, and content management
- **Admin dashboard** with real-time analytics and content editing
- **Custom markdown editor** with live preview and draft/publish workflow
- **Database-driven content** migrated from static markdown files
- **Production deployment** on Vercel with proper environment management

## The Architecture That Emerged

### Database Migration: From Static to Dynamic

One of the most significant decisions was migrating from filesystem-based content to a PostgreSQL database. Instead of maintaining `.md` files scattered across the codebase, I implemented a `Page` model with Prisma:

```typescript
model Page {
  id          String   @id @default(cuid())
  slug        String   @unique
  title       String
  content     String
  isPublished Boolean  @default(false)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

The migration strategy was elegant: a seed script reads existing markdown files and upserts them into the database, preserving all content while enabling dynamic updates.

### The Admin Experience

The admin interface became the heart of the system. Built at `src/app/admin/content/[id]/page.tsx`, it features:

- **Live markdown preview** with split-pane editing
- **Keyboard shortcuts** (Cmd+S for save) that developers expect
- **Draft/publish workflow** with instant toggle
- **Metadata management** for SEO and routing

```typescript
// API route structure that emerged
src/app/api/admin/pages/route.ts        // List and create pages
src/app/api/admin/pages/[id]/route.ts   // Get, update, delete specific pages
```

### Dynamic Content Rendering

All five public content pages now read from the database using a clean abstraction:

```typescript
const content = await getPublishedContent(slug);
if (!content) {
  notFound(); // Built-in Next.js 404 handling
}
```

The `force-dynamic` directive ensures content updates appear immediately, while draft content remains hidden from public visitors.

## Lessons Learned: The Pain Points That Matter

### Environment Variable Gotcha

**The Problem:** After setting up all environment variables on Vercel using heredoc syntax (`<<<`), the entire application broke with "tenant not found" errors.

**The Root Cause:** Every environment variable contained a trailing newline character. `TENANT_SLUG` became `"rd-dasmesser\n"` instead of `"rd-dasmesser"`, causing database queries to fail silently.

**The Solution:** 
```bash
# Wrong approach
vercel env add TENANT_SLUG production <<< "rd-dasmesser"

# Correct approach  
printf 'rd-dasmesser' | npx vercel env add TENANT_SLUG production
```

This taught me that deployment tools can introduce subtle formatting issues that are invisible until everything breaks in production.

### Git Author Authentication

**The Problem:** Vercel deployment failed with "Git author must have access to the team" despite having repository access.

**The Root Cause:** My local Git configuration used my work email (`oliver.baer@jtl-software.com`), but my Vercel account was tied to my personal email (`oliver.baer@gmail.com`).

**The Solution:**
```bash
git config user.email "oliver.baer@gmail.com"
git commit --amend --reset-author
git push --force-with-lease
```

Modern deployment platforms are stricter about identity verification than I expected. The lesson: always align your Git identity with your deployment platform account.

## Key Takeaways for Fellow Developers

Building this marketplace reinforced several important principles:

**Environment management is critical.** Always verify your environment variables don't contain hidden characters. Use `printf` instead of heredoc for sensitive values.

**Identity consistency matters.** Ensure your Git author email matches your deployment platform account to avoid authentication issues.

**Database migration can be seamless.** With proper planning, you can move from static files to dynamic content without losing data or breaking existing functionality.

**Admin interfaces need polish.** Developers using your CMS expect keyboard shortcuts, live previews, and intuitive workflows - don't skimp on the admin experience.
