# Building a Media Upload System: NextAuth Gotchas and Cloudinary Integration

Ever tried to build a complete media management system for your Next.js app? Last week, I tackled exactly that challenge while building an admin dashboard with media uploads, gallery management, and an integrated image picker for content editors. What started as a straightforward feature implementation turned into a masterclass in Next.js 15 authentication quirks and serverless deployment considerations.

## The Vision: Complete Media Management

The goal was ambitious but clear: build a comprehensive media system that would allow administrators to:

- Upload images through a dedicated admin interface
- Browse and manage uploaded media in a gallery view
- Pick and insert images directly into content using a markdown editor
- Store everything efficiently using Cloudinary for hosting and Prisma for metadata

## The Architecture

Here's how I structured the complete system:

### Backend Foundation

First, I extended the Prisma schema with a proper media model:

```prisma
model Media {
  id          String    @id @default(cuid())
  filename    String
  originalName String
  mimeType    String
  size        Int
  url         String
  cloudinaryId String
  type        MediaType
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}

enum MediaType {
  IMAGE
  VIDEO
  DOCUMENT
}
```

The API routes handle the heavy lifting:

- `POST /api/admin/media` - Handles file uploads to Cloudinary
- `GET /api/admin/media` - Lists all media with pagination
- `DELETE /api/admin/media/[id]` - Removes media from both database and Cloudinary

### Frontend Components

The admin interface consists of several key components:

1. **Media Gallery** (`/admin/media`) - Grid view of all uploaded media
2. **Media Picker Dialog** - Modal component for selecting images in editors
3. **Content Editor Integration** - Direct image insertion into markdown content

The magic happens when users click the "Bild" (Image) button in the content editor. It opens the media picker, and selected images get inserted as markdown: `![alt text](image-url)`.

## Lessons Learned: The Hard Way

### NextAuth + Next.js 15: A Silent Killer

The biggest gotcha came from what seemed like a routine authentication check. I was using NextAuth's `getServerSession()` in my admin layout server component:

```tsx
// ❌ This silently fails in Next.js 15
import { getServerSession } from "next-auth/next";

export default async function AdminLayout({ children }) {
  const session = await getServerSession(authOptions);
  // session is always null! 😱
}
```

**The Problem**: `getServerSession()` silently returns `null` in Next.js 15 server components due to incompatibility with the new async `headers()` and `cookies()` functions.

**The Solution**: Switch to JWT token validation:

```tsx
// ✅ This works reliably
import { getToken } from "next-auth/jwt";
import { cookies } from "next/headers";

export default async function AdminLayout({ children }) {
  const token = await getToken({ 
    req: { cookies: cookies() },
    secret: process.env.NEXTAUTH_SECRET 
  });
  
  if (!token) {
    redirect("/admin/login");
  }
  
  return <div>{children}</div>;
}
```

**Key Takeaway**: With NextAuth v4 and Next.js 15, always use `getToken()` in server components. `getServerSession()` will fail silently and waste hours of debugging time.

### Environment Variables and Session Invalidation

Another painful lesson came during Vercel deployment. I had to update `NEXTAUTH_URL` from `http://localhost:3000` to the production URL. Sounds simple, right?

**The Trap**: Changing `NEXTAUTH_URL` invalidates all existing user sessions because it changes the cookie naming scheme:
- HTTP: `next-auth.session-token`
- HTTPS: `__Secure-next-auth.session-token`

Users who were logged in before the change suddenly found themselves locked out, even with valid sessions. The solution was to have them visit `/api/auth/signout` and log back in.

**Pro Tip**: When deploying NextAuth apps, warn users about potential re-login requirements after environment changes.

### Serverless Function Considerations

The media upload initially worked perfectly locally but returned 500 errors on Vercel. This highlighted several serverless-specific considerations:

1. **File Size Limits**: Serverless functions have payload limits
2. **Memory Constraints**: Base64 encoding large images can exceed memory limits  
3. **Timeout Limits**: Cloudinary uploads might timeout on slower connections
4. **Environment Variable Propagation**: Double-check that all Cloudinary credentials made it to production

## The Implementation Details

### Cloudinary Integration

The Cloudinary setup is straightforward but worth showing:

```typescript
// src/lib/cloudinary.ts
import { v2 as cloudinary } from 'cloudinary';

cloudinary.config({
  cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
  api_key: process.env.CLOUDINARY_API_KEY,
  api_secret: process.env.CLOUDINARY_API_SECRET,
});

export async function uploadToCloudinary(file: File) {
  const bytes = await file.arrayBuffer();
  const buffer = Buffer.from(bytes);
  
  return new Promise((resolve, reject) => {
    cloudinary.uploader.upload_stream({
      resource_type: "auto",
      folder: "media",
    }, (error, result) => {
      if (error) reject(error);
      else resolve(result);
    }).end(buffer);
  });
}
```

## Current Status and Next Steps

The system is 90% complete with two remaining issues:

1. **Production Upload Bug**: The media upload POST endpoint returns 500 errors on Vercel (works locally)
2. **Auth Redirect**: Need to add proper redirect logic when admin cookies are invalid

The debugging plan involves checking Vercel function logs and potentially switching from base64 uploads to multipart form data for better serverless compatibility.

## Takeaways for Fellow Developers

1. **Authentication in Next.js 15**: Use `getToken()` from `next-auth/jwt`, not `getServerSession()`
2. **Environment Changes**: Warn users about re-login requirements when changing `NEXTAUTH_URL`
3. **Serverless Uploads**: Test file upload limits early and often on your deployment platform
4. **Media Management**: Separating storage (Cloudinary) from metadata (database) creates a flexible, scalable system

Building a media management system touches every part of your stack - from database design to UI components to serverless deployment constraints. The authentication gotchas were the most surprising part of this journey, but they've made me much more careful about NextAuth version compatibility.
