# Building a Complete Media Management System: From Cloudinary to Markdown Integration

Last week, I tackled one of those features that sounds simple on paper but reveals its complexity once you start digging in: adding a proper media management system to Das Messer CMS. What started as "just add image uploads" evolved into a comprehensive solution involving cloud storage, database modeling, admin interfaces, and seamless editor integration.

## The Vision: More Than Just File Uploads

The goal wasn't just to handle file uploads—it was to create a complete media ecosystem that would feel natural for content creators. I wanted:

- **Drag-and-drop uploads** that feel modern and intuitive
- **Cloud-based storage** with Cloudinary for optimization and CDN delivery  
- **Multi-tenant organization** where each tenant's media stays isolated
- **Visual gallery management** with filtering and search capabilities
- **Seamless editor integration** that makes inserting images effortless

## Architecture Decisions: Building the Foundation

### Database Modeling

First, I extended the Prisma schema with a new `Media` model that captures everything we need to know about uploaded files:

```typescript
enum MediaType {
  IMAGE
  VIDEO
}

model Media {
  id          String    @id @default(cuid())
  filename    String
  originalName String
  mimeType    String
  size        Int
  type        MediaType
  cloudinaryId String
  url         String
  tenantId    String
  tenant      Tenant    @relation(fields: [tenantId], references: [id])
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}
```

This approach gives us rich metadata while maintaining the connection to our multi-tenant architecture. Each media file knows which tenant owns it, enabling proper access control and organization.

### API Design: RESTful and Intuitive

The API surface stayed clean and predictable:

```typescript
// Upload and list media
POST /api/admin/media - FormData upload → Cloudinary → Database
GET  /api/admin/media - Retrieve tenant's media with filtering

// Individual file operations  
DELETE /api/admin/media/[id] - Remove from both Cloudinary and database
```

The upload endpoint handles the complex orchestration: validating file types, uploading to Cloudinary with proper folder structure (`dasmesser/{tenant.slug}/`), and storing metadata in our database—all in a single request.

### Markdown Editor Integration

The real magic happens in the content editor integration. Rather than forcing users to manually type image URLs, I added a visual media picker:

```typescript
const insertImage = (media: Media) => {
  const textarea = textareaRef.current;
  const cursorPos = textarea.selectionStart;
  const textBefore = content.substring(0, cursorPos);
  const textAfter = content.substring(cursorPos);
  
  const imageMarkdown = `![${media.originalName}](${media.url})`;
  const newContent = textBefore + imageMarkdown + textAfter;
  
  setContent(newContent);
  // Focus and position cursor after inserted text
  setTimeout(() => {
    textarea.focus();
    textarea.setSelectionRange(
      cursorPos + imageMarkdown.length, 
      cursorPos + imageMarkdown.length
    );
  }, 0);
};
```

This creates a seamless flow: click "Bild" button → visual media picker opens → click desired image → markdown gets inserted at cursor position → continue writing.

## Lessons Learned: The Devil in the Details

### Environment Variable Management

I hit an unexpected snag when deploying to Vercel. The `vercel env add` commands were failing because I didn't have the Vercel CLI installed globally. A quick `npm i -g vercel` fixed it, but it reminded me how easy it is to assume tools are available.

**Pro tip**: When setting environment variables via CLI, use `printf 'value' | vercel env add VAR_NAME` instead of heredoc syntax to avoid trailing newlines that can break API authentication.

### File Upload Constraints

Setting reasonable limits proved important for both user experience and cost control:

```typescript
const MAX_FILE_SIZE = 10 * 1024 * 1024; // 10MB
const ALLOWED_TYPES = [
  'image/jpeg', 'image/png', 'image/gif', 
  'image/webp', 'image/svg+xml',
  'video/mp4', 'video/webm'
];
```

These constraints prevent abuse while supporting all common web media formats.
