# Migrating From Hardcoded Components to Database-Driven Content: A Hero Block Journey

Sometimes the best refactoring sessions start with a simple goal and end with fixing problems you didn't know existed. Last night was one of those sessions.

## The Mission: One Component to Rule Them All

Our knife crafting website had a problem many developers will recognize: **content duplication**. We had 10 different page components, each with its own hardcoded hero section. Every time we wanted to update hero text, it meant diving into React components instead of using our content management system.

The goal was elegantly simple: create a `:::hero` markdown directive so all 10 pages could use the same generic renderer, with hero content living in the database where it belongs.

## Building the Hero Block System

### Step 1: The Markdown Directive

First, I created a new `HeroBlock.tsx` component that wraps content in a `prose-glass-hero` container:

```tsx
export default function HeroBlock({ children }: { children: React.ReactNode }) {
  return (
    <div className="prose-glass-hero">
      {children}
    </div>
  );
}
```

### Step 2: Parser Integration

The trickier part was extending our markdown parser. I updated `MarkdownContent.tsx` to recognize the new directive:

```tsx
const DIRECTIVE_RE = /^:::(info|warning|hero)\s*$/;

// Added to the Block type union
type Block = InfoBlock | WarningBlock | HeroBlock | TextBlock;

// Extended the parser
if (line.match(/^:::hero\s*$/)) {
  // ... parsing logic
}

// Added to the renderer switch
case 'hero':
  return <HeroBlock key={index}>{children}</HeroBlock>;
```

### Step 3: Styling the Heroes

The CSS was straightforward but important—scaling up typography for impact:

```css
.prose-glass-hero h1 {
  @apply text-4xl md:text-5xl font-bold mb-6;
}

.prose-glass-hero h2 {
  @apply text-3xl md:text-4xl font-semibold mb-4;
}

.prose-glass-hero p {
  @apply text-xl leading-relaxed max-w-3xl;
}
```

### Step 4: The Great Simplification

Here's where it got satisfying. I transformed 10 different page components from this:

```tsx
// Before: Each page had its own hardcoded structure
export default function MaterialPage() {
  return (
    <div className="container mx-auto px-4 py-8 max-w-4xl">
      <Breadcrumbs />
      <section className="text-center mb-12">
        <h1 className="text-4xl font-bold mb-6">Premium Materials</h1>
        <p className="text-xl">Hardcoded hero content...</p>
      </section>
      <MarkdownContent content={content} />
    </div>
  );
}
```

To this:

```tsx
// After: Generic, data-driven structure
export default function MaterialPage() {
  return (
    <div className="container mx-auto px-4 py-8 max-w-7xl">
      <Breadcrumbs />
      <MarkdownContent content={content} />
    </div>
  );
}
```

Clean, consistent, and maintainable.

### Step 5: Database Migration

I wrote a migration script to prepend `:::hero` blocks to all existing content:

```typescript
// prisma/migrate-hero-blocks.ts
const heroBlocks = {
  'material': `:::hero
# Premium Materials
Our knives are crafted from the finest steel...
:::`,
  // ... for all 10 pages
};

for (const [slug, heroContent] of Object.entries(heroBlocks)) {
  await prisma.contentPage.update({
    where: { slug },
    data: { content: `${heroContent}\n\n${existingContent}` }
  });
}
```

## Lessons Learned: When CI Teaches You About Your Codebase

Here's where things got interesting. After pushing the changes, our CI build failed for the first time in... well, the first time I'd seen it succeed, actually.

### The Sitemap Surprise

The error was cryptic: `PrismaClientInitializationError: DATABASE_URL resolved to an empty string`. The culprit? Our `/sitemap.xml` was being statically pre-rendered at build time, trying to connect to a database that didn't exist in the CI environment.

**The fix:** Adding `export const dynamic = 'force-dynamic'` to make the sitemap generate at request time instead of build time.

### The Deeper Database Problem

But wait, there's more! Even after fixing the sitemap, the homepage failed because our header component calls `getMenuItems()` via Prisma. The real issue: we'd never configured the `DATABASE_URL` secret in GitHub Actions.

Rather than just adding the secret, I implemented a more resilient solution:

```typescript
export async function getTenant(): Promise<Tenant | null> {
  if (!process.env.DATABASE_URL) {
    return null; // Graceful degradation for CI environments
  }
  // ... rest of the function
}
```

All our components already handled `null` gracefully, so this fix enabled CI builds without requiring database access.

### The Plot Twist

The best part? After implementing these fixes, I discovered this was the **first successful CI build in the repository's history**. Our refactoring session accidentally fixed a long-standing infrastructure issue!

## The Results

- ✅ 10 pages now use a single, generic component structure
- ✅ Hero content lives in the database where content belongs
- ✅ CI builds are green for the first time ever
- ✅ Code is cleaner and more maintainable
- ✅ Content editors can modify hero sections without touching code

## Takeaways for Fellow Developers

1. **Refactoring often reveals hidden problems**: Our innocent content migration exposed CI issues we didn't know existed.

2. **Graceful degradation is powerful**: Instead of requiring perfect environment configuration, design your functions to handle missing dependencies elegantly.

3. **Migration scripts should be idempotent**: Always check if changes have already been applied before making database modifications.

4. **Sometimes the best fixes are the simplest ones**: A single `if` statement solved our CI problem more elegantly than complex environment configuration.

The session wrapped up around 21:30 UTC with three commits pushed and a much cleaner codebase. Sometimes the best development sessions are the ones where you set out to solve one problem and accidentally fix several others along the way.
