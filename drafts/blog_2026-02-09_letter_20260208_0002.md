# Building a Content Management System: From File-Based to Database-Driven with Real-Time Editing

Last week, I set out to transform a simple landing page into something more dynamic. What started as "just add a markdown editor" turned into a full migration from file-based content to a database-driven CMS with draft/publish workflows. Here's how it went down, including the deployment headaches that made me question my life choices.

## The Vision: From Static to Dynamic

The goal was straightforward: take our landing page for Das Messer (a knife marketplace) and add content management capabilities. No more pushing code changes just to update a product description or fix a typo. The plan included:

- **Database migration**: Move content from `.md` files to PostgreSQL
- **Admin interface**: Build a markdown editor with live preview
- **Draft/publish workflow**: Let content creators save drafts before going live
- **Seamless deployment**: Make it all work on Vercel without breaking existing URLs

Sounds simple, right? *Narrator: It was not simple.*

## The Technical Journey

### Step 1: Database Migration

First challenge was migrating existing content from the filesystem to PostgreSQL. Rather than manually copying content, I updated the Prisma seed script to read the existing `.md` files and upsert them into a new `Page` table:

```typescript
// prisma/seed.ts
const pages = [
  { slug: 'material', route: '/material' },
  { slug: 'herstellung', route: '/herstellung' },
  { slug: 'schleifen', route: '/schleifen' },
  // ... more pages
];

for (const page of pages) {
  const content = fs.readFileSync(`content/${page.slug}.md`, 'utf-8');
  await prisma.page.upsert({
    where: { slug: page.slug },
    update: { content },
    create: {
      slug: page.slug,
      route: page.route,
      title: extractTitle(content),
      content,
      published: true,
      tenantId: tenant.id
    }
  });
}
```

This approach preserved all existing content while setting up the database structure for future dynamic updates.

### Step 2: API Layer Design

The content API needed to handle both public consumption and admin operations. I split this into separate concerns:

```typescript
// Public content access
export async function getPublishedContent(slug: string) {
  const page = await prisma.page.findFirst({
    where: { slug, published: true }
  });
  return page;
}

// Admin API routes
// GET /api/admin/pages - List all pages
// POST /api/admin/pages - Create new page
// PUT /api/admin/pages/[id] - Update existing page
```

The key insight here was maintaining backward compatibility. All existing public routes continued to work exactly as before, but now pulled content from the database instead of files.

### Step 4: Draft/Publish Workflow

This was crucial for content creators who need to iterate before publishing. The implementation tracks publication status at the database level:

```typescript
// Only published content appears on public routes
export async function getPublishedContent(slug: string) {
  const page = await prisma.page.findFirst({
    where: { 
      slug, 
      published: true,  // This is the key filter
      tenant: { slug: process.env.TENANT_SLUG }
    }
  });
  
  if (!page) {
    notFound(); // Returns 404 for drafts on public routes
  }
  
  return page;
}
```

Content creators can save drafts as often as needed, preview them in the admin interface, and publish when ready.

## Lessons Learned: Deployment Gotchas

The technical implementation went smoothly. Deployment? That's where things got spicy.

### Challenge 1: The Mysterious Trailing Newlines

I was setting Vercel environment variables using heredoc syntax:

```bash
vercel env add TENANT_SLUG production <<< "rd-dasmesser"
```

Everything looked fine until the app couldn't find the tenant in production. Debugging revealed that all environment variables had literal `\n` characters appended:

```
TENANT_SLUG="rd-dasmesser\n"
```

The database lookup failed because `"rd-dasmesser"` doesn't match `"rd-dasmesser\n"`.

**Solution**: Use `printf` to avoid trailing newlines:

```bash
printf 'rd-dasmesser' | vercel env add TENANT_SLUG production
```

### Challenge 2: Git Author Permissions

Vercel deployment failed with a cryptic error: "Git author must have access to the team." Turns out my local git was configured with my work email, but the Vercel project was under my personal account.

**Solution**: Update local git config and amend the commit:

```bash
git config user.email "oliver.baer@gmail.com"
git commit --amend --reset-author --no-edit
git push --force-with-lease
```

## Key Takeaways

**For environment variables**: Always verify what's actually being set. Invisible characters like trailing newlines can cause mysterious failures in production.

**For git workflows**: Keep your commit author email aligned with your deployment platform's account. It seems obvious in retrospect, but it's easy to miss when switching between personal and work projects.

**For CMS migrations**: Maintain backward compatibility during transitions. Users shouldn't notice when you flip the switch from static to dynamic.

**For developer experience**: Small touches like Cmd+S shortcuts and live preview make the difference between a tool people tolerate and one they actually enjoy using.

Building a CMS from scratch reminded me why tools like WordPress and Contentful exist—there are so many details to get right. But for specific use cases, a custom solution can provide exactly the workflow your team needs without the bloat of generic solutions.

The code is running smoothly in production, and the content team is already using the editor to iterate on copy. Sometimes the best measure of success is when the tool disappears into the background and people can focus on their actual work.
