# Building a Dynamic CMS: From Static Pages to Database-Driven Content

Ever started with a simple static site and found yourself thinking, "What if I could edit this content without touching code?" That's exactly what happened during my latest development session, where I transformed a static Next.js landing page into a full-featured content management system.

## The Vision

The goal was ambitious but clear: build an admin interface that would allow non-technical users to:
- Configure site settings and toggle dark mode
- Manage dynamic navigation menus
- Edit page content with enhanced markdown capabilities
- All while maintaining the site's visual appeal with background images and glass-card effects

## What Got Built

### 🗄️ Database Foundation

First, I extended the Prisma schema with two key models:

```typescript
model SiteSettings {
  id       String  @id @default(cuid())
  siteName String?
  logoUrl  String?
  darkMode Boolean @default(false)
}

model MenuItem {
  id       String @id @default(cuid())
  title    String
  href     String
  order    Int
  isActive Boolean @default(true)
}
```

These simple models became the backbone for site-wide configuration and dynamic menu generation.

### 🎨 Dark Mode Implementation

Implementing dark mode properly required coordination across multiple layers:

- **Tailwind Configuration**: Switched to class-based dark mode strategy
- **CSS Variables**: Added dark variants in `globals.css`
- **Anti-Flash Script**: Prevented the dreaded white flash on page load
- **Component Integration**: Built a toggle component that persists preferences

The trickiest part was preventing the flash of wrong-theme content (FOIT). The solution was injecting a blocking script directly into the document head that checks localStorage before the page renders.

### 📝 Advanced Markdown Processing

Here's where things got interesting. The existing markdown parser couldn't handle nested directives like:

```markdown
:::columns
Left column content

:::box type="info"
This is a nested box inside a column
:::

---

Right column with **bold text**
:::
```

I rewrote the parser with a depth-aware algorithm that:
1. **Tracks nesting levels** using delimiter counting
2. **Recursively processes blocks** to handle arbitrary nesting
3. **Preserves column splitting** even within nested structures
4. **Supports image sizing** with `![alt|600x400](url)` syntax

The breakthrough was realizing that each `:::` level needed its own parsing context, leading to a recursive `RenderBlocks` component that could handle infinite nesting.

### 🖼️ Visual Polish

The glass-card transparency effect looked great in the hero section, but fell apart when content sections had opaque backgrounds. The fix was simple but impactful: wrap the entire page (not just the hero) with the background image container, then use transparency gradients to let the background show through.

## Lessons Learned (The Hard Way)

### The `.next` Cache Trap

**Problem**: After making extensive file changes, the dev server threw `ENOENT: .next/server/vendor-chunks/next-auth.js` errors.

**Solution**: Always `rm -rf .next` after major structural changes. Next.js's cache can become stale and cause phantom module resolution errors.

### Serverless Cold Starts

**Problem**: `PrismaClientInitializationError: Can't reach database server` on fresh deployments.

**Reality Check**: This wasn't a bug—it's just how serverless databases work. Neon DB connections take a moment to warm up from a cold start. The solution is patience (and proper error handling).

### Shell Globbing Gotchas

**Problem**: Git commands failed with paths like `src/app/admin/content/[id]/page.tsx`.

**Solution**: Quote paths containing special characters: `"src/app/admin/content/[id]/page.tsx"`. Zsh's aggressive globbing can trip you up with Next.js dynamic routes.

## The Result

What started as a static landing page is now a fully dynamic CMS where content editors can:
- Toggle dark/light themes site-wide
- Rearrange navigation menus via drag-and-drop
- Write rich content using nested markdown directives
- Preview changes in real-time
- Manage multiple pages from a central dashboard

The "Über uns" (About Us) page was the first to be migrated from static JSX to database-driven content, using the new nested markdown syntax to recreate complex layouts that previously required custom components.

## What's Next

The foundation is solid, but there's more to explore:
- Converting the homepage to editable content
- Adding more background image options through the admin
- Building a media library for easier image management
- Implementing content versioning and drafts

## Technical Stack

- **Framework**: Next.js 14 with App Router
- **Database**: Neon PostgreSQL with Prisma ORM
- **Styling**: Tailwind CSS with custom dark mode
- **Deployment**: Vercel with automatic deployments
- **Authentication**: NextAuth.js for admin access

The beauty of this approach is that it maintains the performance benefits of static generation while providing the flexibility of dynamic content management. Each page can be statically generated at build time, yet updated through the admin interface whenever needed.
